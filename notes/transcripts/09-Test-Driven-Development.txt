Test driven development is a method of writing code that involves writing automated tests first
and then developing code to pass those tests. It's widely adopted in professional programming
circles and can help lead to fewer bugs being shipped in your code. Now that we have the basics
of embedded rust out of the way, we can start looking at how to apply those concepts. We'll
start by seeing how to use test driven development for embedded code. This can be a little tricky
as it often means writing dummy or stub implementations of protocols so we can create automated tests.
I'll show you how to create a stub I squared C implementation so we can create automated unit
tests for our TMP 102 driver. We wrote a few episodes ago. The basic idea behind test driven development
is that you write your tests before you write your functional code in the first phase often called
the red phase. You create a unit test around some feature request or bug you wish to fix. You sometimes
need to write just enough functional code that the test runs, but it should be failing at this point.
You then write functional code such that the test passes. This is known as the green phase. Your code
might not be fully optimized. So then you move into the refactoring phase where you rewrite your functional
code in an optimized way. This is sometimes called the blue phase. The refactoring process will sometimes
introduce new bugs or you'll often need to tackle a new feature request. So you then do the cycle again, adding a
test, writing functional code until it passes and refactoring your code. You'll often see tests broken out into three
categories. The first is unit tests. These are usually simple automated tests that verify the correct
operation of individual values, methods or functions in your code. Next are integration tests where you
verify that larger components operate together. Then you have system or end to end tests that verify that
everything works together. This is usually where you test the entire application and it can often be quite
time consuming depending on the complexity of your program. As we'll see unit tests are usually
possible in embedded code, but integration and system tests become much trickier as you either need to
automate testing with real hardware known as hardware in the loop, or you have to find ways to emulate the
functionality of low level drivers like GPIO and I squared C. If we're testing our TMP 102 driver, we'll need to emulate the
I squared C driver. Since we won't be using hardware in the loop, you'll see the following types of
substitutions used quite often in test driven development. The definitions can vary depending on who
you ask. But here are some of the terms you'll come across. The first is a dummy. This is when something in
the code like an object, struct or value is passed around, but doesn't actually do anything. It's just there to fill out
parameters in say function calls. The next is a fake. These are objects or structs that have mostly working
implementations, but they can take shortcuts which make them not really suitable for production. Then you
have a stub. These are bare bones implementations of classes, structs or functions that provide canned
predetermined responses to calls. They usually do not respond to anything other than what's needed for the tests.
Sometimes you'll see the term spy, which is similar to a stub, but it records information on how the function
or class was called or used. Mocks are similar to stubs, but they offer more implementation under the hood.
They usually perform some internal checks to ensure that the received values or calls were performed
correctly and they are capable of failing a test from within the implementation. As I mentioned,
these definitions can differ depending on who you ask and they're often used interchangeably with each
other. We'll be mostly using dummies and stubs to simulate I squared C calls to get our TMP 102 driver
to work without needing real hardware. By building tests around your functional code, you can easily
automate tests to verify your work every time you make updates. This is a crucial step in the continuous
integration process, which is half of the continuous integration and continuous delivery or CICD pipeline.
Even in the embedded world, setting up such a pipeline can save you a lot of headaches by catching bugs early.
