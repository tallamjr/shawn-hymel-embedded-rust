In general, most programming languages make a trade-off
between two different memory management strategies.
The first is manual memory management.
Older, lower-level languages like C and C++
force users to allocate and deallocate memory manually.
On the plus side, this usually means faster runtimes
as there is no overhead for these kinds of operations.
It also means predictable performance
as what you write is what gets executed
with some assumed compiler optimizations.
It also means that your program sizes could be quite small
as there was no extra overhead for memory cleanup.
This is why such languages, specifically C,
are the most common in embedded systems.
However, programmers need to know exactly what they're doing
as it's easy to create memory leaks
or undefined behaviors in programs.
To afford this kind of low-level control,
such languages are often quite verbose,
and developers need to track and debug memory allocations
and deallocations, adding to development time and complexity.
This means writing such programs can be slow and painstaking.
Higher-level languages like Python, Java, and JavaScript
attempt to address these potential memory bugs
by adding something called the garbage collector.
This is a system that runs automatically with your code
to track memory allocations
and freeze up any memory that is no longer used.
It's usually implemented as a separate thread

It promises much better safety

Languages that use a garbage collector

so developers can focus on higher-level concepts.
This means that such automatic cleanup
can afford faster development times.

The garbage collector has overhead,

which some programmers often want
